根据您的要求，所有预热子模块均为必选项，不再设置“可选”或“跳过”的降级逻辑，但允许因浏览器/硬件不支持而采用回退机制，确保代码仍能运行且不报错。同时，保留用户可选的性能模式（极致性能、均衡、节能），用于控制预热强度，而非决定是否执行。

以下是修改后的需求模块，整合到原文档第 4 章“底层架构与生存优化”中。

## 4. 底层架构与生存优化（补充）

### 4.6 极速启动与硬件资源预热模块

#### 4.6.1 设计目标
- **零感知预热**：页面加载时立即无感调用所有可用硬件资源（CPU、GPU、内存、网络、存储、音频、视频编码、神经计算单元等），使设备在用户首次交互前即进入高性能状态。
- **无额外授权**：所有预热操作均不触发浏览器权限弹窗，不涉及摄像头、麦克风、地理位置、蓝牙等敏感 API，完全基于 Web 标准允许的默认能力。
- **极致性能**：通过预分配、预连接、预编译、预运行等手段，消除后续模拟运行时的首次加载延迟、JIT 编译开销、资源申请抖动，确保文明模拟在首帧即达到目标帧率。
- **用户可控的性能模式**：在设置面板中提供“性能模式”选项，允许用户选择“极致性能”“均衡模式”“节能模式”，不同模式下预热强度与资源占用策略不同，满足不同场景需求。
- **必选性**：所有预热子模块均需实现，不因浏览器/硬件支持情况而跳过；对于无法直接使用的 API，必须执行回退或模拟操作，保证代码路径始终被执行，同时避免报错。

#### 4.6.2 预热资源清单与技术要求（必选实现）

| 资源类型 | 预热手段 | 技术约束与回退方案 | 对用户感知的影响 |
|----------|----------|---------------------|------------------|
| **CPU 多核** | 根据 `navigator.hardwareConcurrency` 创建等量的 Web Worker，每个 Worker 中实例化核心 Wasm 模块并执行极短时间的空计算（<5ms），触发 CPU 升频与线程池激活。 | 需配置 COOP/COEP 头以支持 SharedArrayBuffer 和多线程。若 Worker 创建失败或 Wasm 实例化失败，则至少创建 2 个 Worker 并运行纯 JavaScript 空循环作为回退。 | 无 UI 卡顿，Worker 在后台静默运行。 |
| **GPU** | 创建隐藏的 OffscreenCanvas（1x1 或极小尺寸），初始化 WebGPU 设备，提交一个空计算着色器或简单渲染任务，强制 GPU 驱动进入高频率状态。 | 若 WebGPU 不可用，回退到 WebGL 2.0 的相似预热（创建 WebGL 上下文，绘制一个像素）。若 WebGL 亦不可用，则使用 Canvas 2D API 触发 GPU 合成。 | 不显示任何内容，不占用可见区域。 |
| **内存** | 页面加载时立即分配大块 SharedArrayBuffer，并用原子操作填充一次，确保内存页被实际映射并预热。 | 若 SharedArrayBuffer 因 COOP/COEP 未配置而不可用，则分配普通 ArrayBuffer 作为替代，并在控制台输出警告（仅调试模式）。 | 无感知。 |
| **网络** | 使用 WebTransport（或备选 WebRTC）预先与对等节点建立 QUIC 连接，不发送业务数据，仅完成握手和会话保持。 | 若 WebTransport 不可用或连接失败，回退到 WebRTC 数据通道（静默建立连接）；若仍失败，则发起一次简单的 `fetch` 请求（指向自身或健康检查端点）预热网络栈。 | 无感知。 |
| **存储** | 使用 OPFS 打开默认数据库文件，创建 SQLite Wasm 实例并执行一次 `SELECT 1` 查询，预热文件系统缓存和 SQLite 引擎。 | 若 OPFS 不可用，回退到 IndexedDB 打开一个数据库并执行一次空读写；若 SQLite Wasm 初始化失败，则使用原生 IndexedDB 操作作为替代。 | 无感知。 |
| **音频** | 创建 AudioContext 并立即挂起（`suspend()`），初始化音频图但不播放任何声音，使音频线程启动并保持就绪。 | 若 AudioContext 创建失败（如被浏览器限制），则使用 `OfflineAudioContext` 作为回退，同样进行初始化但不渲染。 | 无声音输出。 |
| **视频编码** | 创建 VideoEncoder 实例，配置一种硬件支持的编码器（如 `vp8`、`h264`），传入一个空帧并立即关闭，促使硬件编码器加载驱动。 | 若 VideoEncoder 不可用或配置失败，则尝试使用 `MediaRecorder` 初始化一个空流作为回退；若仍失败，则跳过具体操作但保留代码调用路径（调用空函数）。 | 无感知。 |
| **神经计算** | 若 WebNN API 可用，创建 MLContext，执行一次最简单的推理（如单层全连接），预热 NPU 或 GPU 神经网络路径。 | 若 WebNN 不可用，则使用 TensorFlow.js 或 ONNX Runtime Web 的轻量级模型（如单层线性回归）执行一次推理，确保神经网络相关代码路径被预热。若以上均不可用，则运行纯 JavaScript 的简单矩阵乘法模拟。 | 无感知。 |
| **Wasm 模块** | 在多个 Worker 中提前实例化核心 Wasm 模块（量子模拟、SNN、八字引擎等），保持实例引用，避免首次使用时加载和编译延迟。 | 若 Wasm 流式编译失败，则使用 `WebAssembly.instantiate` 从 ArrayBuffer 实例化；若整体 Wasm 不支持，则使用 JavaScript 备用实现（需提前声明，确保核心功能可用）。 | 无感知。 |

#### 4.6.3 性能模式选项与参数控制
在游戏“设置”面板中增加“性能模式”下拉选项，包含以下三种模式：

- **极致性能模式**  
  - 启用所有预热资源，使用最高规格参数：  
    - 内存分配：512MB（SharedArrayBuffer）  
    - Worker 数量：`navigator.hardwareConcurrency`（上限 16）  
    - 所有回退路径均尝试最高性能配置  
  - 不设电量、温度保护，以榨干设备性能为首要目标。  

- **均衡模式**（默认）  
  - 启用所有预热资源，使用适中参数：  
    - 内存分配：256MB  
    - Worker 数量：`Math.max(2, Math.floor(navigator.hardwareConcurrency / 2))`  
  - 保证流畅模拟，适度控制功耗与内存占用。  

- **节能模式**  
  - 启用所有预热资源，但使用低强度参数：  
    - 内存分配：128MB  
    - Worker 数量：2  
    - 对于可选的高功耗预热（如 WebTransport 保持连接、神经计算）仍执行，但降低并发或减少预热时长（如连接后立即断开）。  

用户选择性能模式后，预热模块根据所选模式调整参数。所有预热子模块的代码路径均被触发，只是资源占用强度不同。

#### 4.6.4 启动流程与时间点
- **加载阶段（HTML 解析时）**  
  - 读取用户上次保存的性能模式设置（若无则使用均衡模式）。  
  - 同步执行最小的预热脚本（内联），立即根据所选模式分配 SharedArrayBuffer、启动对应数量的 Worker、初始化隐藏 canvas 和 WebGPU。  
  - 不等待 DOMContentLoaded。

- **页面可见后（用户未交互）**  
  - 根据性能模式完成剩余预热（WebTransport、OPFS、AudioContext、WebCodecs、WebNN 等）。  

- **用户首次交互前**  
  - 所有预热操作（含回退路径）均已完成，文明模拟主循环可立即以最高性能启动。

#### 4.6.5 合规性与透明性
- **不请求额外权限**：所有预热 API 均属于浏览器默认允许的能力，无需用户授权。AudioContext 仅创建不播放；WebTransport 仅连接不传输用户数据；WebCodecs 不读取摄像头/屏幕；WebNN 仅本地推理。
- **用户告知**：在游戏“关于”或“性能”页面中，增加说明：“本游戏可根据用户选择的性能模式，自动调用设备硬件资源以获得最佳体验，不收集任何个人信息。” 作为透明披露。
- **隐私保护**：预热过程中不读取任何设备指纹（WebGL 信息、canvas 指纹等）并上传，仅本地使用。

#### 4.6.6 性能验收指标
- 在极致性能模式下，从页面加载到模拟首帧输出 ≤ 2 秒，且首帧帧率达到目标（30fps）的 90% 以上。
- 在均衡模式下，预热过程不造成主线程明显卡顿，内存占用控制在合理范围。
- 所有预热子模块均执行，即使某一模块回退，游戏仍能正常启动，并在控制台输出降级日志（仅调试模式可见）。

#### 4.6.7 与现有模块的集成
- 预热模块的结果（如预分配的 SharedArrayBuffer、预实例化的 Wasm 实例）直接供给“底层架构与生存优化”中的零拷贝架构、SharedWorker 内核使用。
- 预热模块的生命周期由“初始化管理器”统一协调，保证各模块启动顺序与依赖关系。
- 性能模式的选择结果应持久化存储（使用 IndexedDB 或 localStorage），以便用户下次打开时沿用。

## 4.7 其他极致性能优化措施（必选）

### 4.7.1 代码与资源加载优化
- **关键资源预加载**  
  - 使用 `<link rel="preload">` 预加载核心 Wasm 模块、关键字体、首屏所需的 WebGPU 着色器代码。  
  - 使用 `<link rel="preconnect">` 提前与 CDN 及 WebTransport 服务器建立连接。  
  - 使用 `<link rel="dns-prefetch">` 提前解析外部域名。  
- **资源压缩与格式优化**  
  - 所有文本资源（JS、CSS、HTML）启用 Brotli 或 gzip 压缩。  
  - 图片使用 AVIF 或 WebP 格式，图标使用 SVG 或字体图标。  
  - 字体文件使用 WOFF2 格式，并利用 `font-display: swap` 避免阻塞渲染。  
- **代码分割与按需加载**  
  - 使用 Vite 的动态导入（`import()`）将非首屏模块（如设置面板、成就系统、跨文明交互）拆分为独立 chunk，仅在需要时加载。  
  - 核心模拟循环代码在首屏同步加载，其余功能通过 `requestIdleCallback` 延迟加载。

### 4.7.2 渲染与动画优化
- **硬件加速合成**  
  - 所有动画使用 `transform` 和 `opacity` 属性，避免触发重排。  
  - 对频繁变化的元素使用 `will-change` 提示浏览器。  
- **离屏渲染**  
  - 使用 `OffscreenCanvas` 在 Worker 中执行复杂渲染（如 3D 高斯溅射、汉字粒子系统），避免阻塞主线程。  
- **View Transitions 原生动画**  
  - 界面切换统一使用 View Transitions API，由浏览器合成器执行，减少主线程负担。  
- **CSS 内容优先**  
  - 首屏关键样式内联，非关键样式异步加载。  
  - 使用 CSS 网格或 Flexbox 快速布局，避免复杂 JavaScript 计算位置。

### 4.7.3 计算与线程调度优化
- **多线程全量利用**  
  - 除预热 Worker 外，将文明模拟的核心任务（市民 SNN 更新、量子电路模拟、经济模型计算）全部分配到 Worker 池中，主线程仅负责 UI 渲染和用户输入。  
  - 使用 `SharedWorker` 确保多 Tab 共享同一模拟实例，减少重复计算。  
- **WebGPU 计算着色器批处理**  
  - 将大量同质化计算（如粒子运动、市民健康值更新、情感网络传播）全部迁移至 WebGPU 计算着色器，利用 GPU 并行能力。  
  - 设计紧凑的缓冲区布局，减少 CPU-GPU 数据传输。  
- **Wasm 模块极致优化**  
  - 核心数学函数（如向量运算、矩阵乘法）使用 SIMD 指令集（Wasm SIMD）加速。  
  - 编译时启用最高优化等级（`-O3`），并去除所有调试信息。  
- **零拷贝数据流转**  
  - 所有跨线程、跨 Worker 的大数据（如市民状态数组、粒子位置）均通过 `SharedArrayBuffer` 传递，避免序列化开销。  
  - 使用原子操作进行同步，确保数据一致性。

### 4.7.4 存储与网络优化
- **OPFS + SQLite 高频查询优化**  
  - 为常用查询（如市民排行榜、历史事件检索）创建索引，预热时执行一次查询以触发索引加载。  
  - 使用内存模式（`:memory:`）的 SQLite 实例存储临时数据，减少磁盘 I/O。  
- **Service Worker 预缓存**  
  - 注册 Service Worker，在安装阶段预缓存所有核心资源（HTML、JS、Wasm、字体、关键图片），实现离线启动。  
  - 采用 Cache-First 策略，确保二次加载瞬间完成。  
- **WebTransport 长连接保持**  
  - 在极致性能模式下，保持 WebTransport 连接持续活跃，定期发送空 ping 帧，防止空闲超时。  
- **HTTP/3 与 QUIC**  
  - 服务器配置支持 HTTP/3，利用 QUIC 的多路复用和 0-RTT 特性，减少连接建立时间。

### 4.7.5 硬件特性深度利用
- **WebNN 神经计算单元**  
  - 对于支持 WebNN 的设备，将市民的 SNN 推理、LLM 的轻量级任务（如情绪分类）卸载到 NPU，释放 CPU/GPU 负载。  
- **WebCodecs 硬件编码加速**  
  - 录制功能使用硬件编码器，边玩边录时不影响主循环帧率。  
- **WebXR 与 WebGPU 协同**  
  - 若用户连接 VR/AR 设备，使用 WebXR 的 GPU 层直通，避免额外合成开销。  
- **Web Bluetooth 近场通信**  
  - 仅用于增强社交体验，不涉及隐私数据，且在用户主动操作前不主动扫描。

### 4.7.6 首屏体验与感知优化
- **骨架屏与进度反馈**  
  - 在预热和加载期间显示骨架屏，并精确反馈各项资源加载进度（Wasm、WebGPU、数据库等），让用户感知到“正在全力准备”。  
- **渐进式交互**  
  - 首帧渲染完成后，立即响应用户的缩放、点击操作，避免因后台计算导致界面卡顿。  
- **性能监控面板**  
  - 提供可选的实时性能面板（FPS、内存、GPU 占用），供高级用户评估设备负载，但不影响普通用户。

### 4.7.7 与性能模式的联动
上述所有优化措施均受用户选择的性能模式影响：
- **极致性能模式**：启用全部优化，无限制占用资源（如保持 WebTransport 长连接、全量 Worker 运行、GPU 常驻高负载）。  
- **均衡模式**：启用大部分优化，但限制后台连接数量和 Worker 数量，避免过度耗电。  
- **节能模式**：仅启用对性能影响最大的优化（如 Wasm、WebGPU 计算），关闭网络长连接和部分 Worker，确保核心模拟流畅。

### 4.7.8 合规性重申
所有优化措施均不请求额外权限，不读取用户隐私数据，不进行设备指纹收集。Service Worker 仅缓存静态资源，不拦截用户数据。WebTransport 连接仅用于多文明同步，不传输个人信息。

**说明**：以上措施将作为现有预热模块的补充，形成完整的“极致性能启动与运行”需求体系，确保《永夜熵纪》在打开瞬间即调动全部硬件潜能，并持续以最高效率运行。


为满足“网站将成为测试电脑性能的基准工具”这一目标，需在现有性能优化基础上进一步强化，将硬件资源压榨到极限，并提供可重复、可量化的测试能力。以下需求补充至原文档，作为独立模块或集成至已有章节，确保在不额外索权的前提下，实现极致性能测试。

## 3.27 性能基准测试模块（新增）

### 3.27.1 概述
- **定位**：内置专业级性能基准测试工具，允许用户在“极致性能模式”下运行标准化测试，量化评估设备的计算、图形、内存、存储、网络等综合性能。
- **目的**：为硬件评测、超频验证、散热测试提供真实负载；同时作为游戏内“性能认证”功能，生成设备性能评分与排行。

### 3.27.2 测试模式与触发方式
- **独立测试入口**：游戏主菜单增加“性能基准测试”按钮，点击后进入测试界面，可选择测试项（全量/单项）和测试时长（快速/标准/极限）。
- **后台静默测试**：在用户开启“极致性能模式”并运行游戏时，系统持续记录性能指标（FPS、帧时间、内存占用、GPU 负载等），形成长期性能日志，供用户导出分析。

### 3.27.3 测试项目与指标

| 测试类别 | 测试内容 | 测量指标 | 实现方式 |
|----------|----------|----------|----------|
| **CPU 多核** | 大规模市民 SNN 更新（10万+ 神经元）、量子电路模拟（16 qubits）、经济模型迭代 | 每秒模拟步数、平均帧时间、多线程加速比 | 利用所有 Worker 满载运行，禁用任何节流 |

| **内存带宽** | 大规模 SharedArrayBuffer 读写、市民状态批量拷贝 | 内存分配/释放速率、原子操作吞吐 | 分配超大缓冲区，反复读写并计时 |
| **存储 I/O** | OPFS + SQLite 批量写入（10万条记录）、索引查询 | 写入/查询耗时、IOPS | 模拟游戏内高频写入场景，记录时间 |
| **网络延迟** | WebTransport 数据报往返延迟、吞吐量测试 | 最小/平均/最大延迟、带宽 | 向测试服务器发送大小不同的数据包，统计往返 |
| **WebAssembly** | Wasm 模块密集计算（矩阵乘法、傅里叶变换） | 计算耗时、调用频率 | 独立 Wasm 函数重复执行，记录总时间 |
| **神经计算** | WebNN 推理吞吐量（若支持） | 推理延迟、功耗估算 | 运行标准模型（如 MobileNet）100次 |

### 3.27.4 测试执行策略
- **可重复性**：每次测试前清理缓存，重置状态，确保测试环境一致。
- **热身阶段**：正式测试前运行 5 秒预热，让设备进入稳定高频状态。
- **持续时间**：用户可选 30 秒（快速）、60 秒（标准）、120 秒（极限）或“无限直至手动停止”。
- **负载极限**：在极致性能模式下，测试负载应推至硬件极限，直至触发系统降频、温度墙或浏览器崩溃（记录崩溃点）。测试结束后报告“最高稳定性能”与“极限崩溃阈值”。

### 3.27.5 结果展示与导出
- **实时监控**：测试过程中显示实时 FPS、CPU 温度（若 API 可用）、内存占用、GPU 负载图表。
- **最终报告**：生成包含各项分数、百分位数、设备信息（浏览器、操作系统、GPU 型号等）的详细报告。
- **导出格式**：JSON、CSV 或可读 HTML 报告，支持分享。
- **全球排行榜**（可选）：用户可自愿上传测试结果（匿名），形成性能排行榜，用于横向对比。

### 3.27.6 性能模式联动
- 基准测试仅在“极致性能模式”下启用全部测试项；若用户选择其他模式，则仅提供受限测试或提示切换模式。
- 测试期间自动临时屏蔽节电策略（如 `requestAnimationFrame` 节流、后台标签页降频），确保测试准确。

### 3.27.7 稳定性与合规
- **超时保护**：单项测试超过 30 秒无响应则自动终止，避免浏览器假死。
- **用户控制**：测试可随时中止，恢复游戏正常运行。
- **不额外索权**：所有测试不请求摄像头、麦克风、位置等权限；温度读取仅限浏览器暴露的 `navigator.getBattery().temperature` 等非标准 API（若存在），不存在则跳过。
- **隐私**：设备信息仅用于测试报告，不上传除非用户明确同意。

## 4. 底层架构与生存优化（再补充）

### 4.8 极限性能压榨措施（性能测试场景专用）

为达到“拉满”目标，在极致性能模式下需额外实施以下激进优化：

#### 4.8.1 资源无节制抢占
- **Worker 数量**：创建 `navigator.hardwareConcurrency` 个 Worker，并为每个 Worker 分配独立 Wasm 实例，满载运行计算任务。
- **内存分配**：分配接近浏览器单标签页内存上限（通常 2GB~4GB）的 SharedArrayBuffer，并持续读写以触发内存带宽极限。
- **GPU 常驻高负载**：WebGPU 渲染循环不设帧率上限，持续提交复杂计算着色器，使 GPU 利用率保持在 99% 以上。

#### 4.8.2 消除所有节流机制
- **禁用 `requestAnimationFrame` 节流**：当页面不可见时，浏览器默认会降低帧率。在极致性能模式下，通过创建隐藏的 `OffscreenCanvas` 并持续请求渲染，维持高帧率（即使标签页非激活）。
- **禁用后台限流**：使用 `setInterval` 或 `postMessage` 循环唤醒 Worker，阻止浏览器将后台标签页降频。

#### 4.8.3 模拟硬件中断
- **不间断计算**：在主线程和 Worker 中运行无条件的 `while` 循环，仅通过 `Atomics.wait` 或 `yield` 短暂让出控制权，确保 CPU 始终满载。
- **GPU 同步等待**：使用 `GPUQueue.onSubmittedWorkDone()` 强制 CPU 等待 GPU 完成，消除流水线空闲。

#### 4.8.4 极速 I/O 压测
- **OPFS 并发写入**：同时从多个 Worker 对同一数据库文件进行高并发写入，测试存储系统极限。
- **网络洪泛**：持续向测试服务器发送 WebTransport 数据报，填满上行带宽。

#### 4.8.5 崩溃与稳定性测试
- **逐步加压**：自动递增负载（如每 10 秒增加 20% 市民数量），直至浏览器崩溃或系统卡死，记录崩溃时的负载水平作为设备“性能上限”。
- **错误恢复**：崩溃后自动重启测试，并提示用户降低负载或切换模式。

### 4.9 性能监控与反馈系统

#### 4.9.1 实时性能仪表盘
- 在游戏界面（可开启/关闭）显示实时 FPS、帧时间、内存占用、GPU 利用率、网络往返延迟、存储写入速率等。
- 使用图表或数字形式，颜色编码（绿色正常，黄色警告，红色过载）。

#### 4.9.2 性能日志与历史记录
- 自动记录每次游戏会话的性能数据，存储于 IndexedDB，支持导出。
- 用户可查看历史性能趋势，了解设备长期稳定性。

#### 4.9.3 性能警告与建议
- 当检测到频繁掉帧、内存接近上限、温度过高（若可检测）时，主动提示用户切换至均衡模式或节能模式。
- 提供“一键优化”建议，如关闭其他标签页、开启硬件加速等。
